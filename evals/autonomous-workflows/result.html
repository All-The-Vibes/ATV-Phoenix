Ralph loop driver · dist/ralph/phoenix-ralph.ps1 (no real Copilot calls — stubbed agent)

CASE A — vacuous top-level gate refused
[ralph] baseline sense of done-check...
[ralph] FATAL: done-check is ALREADY GREEN at start --
        it can't prove failure-first (it'd be a vacuous gate).
driver exit=2   (refuses to start on a gate that can't fail)
CASE B — happy path, completion proven
[ralph] iteration 1/5 -- invoking agent (fresh context)...
[ralph] done-check ACCEPTED (failure-first, green).
[ralph] COMPLETE in 2 iterations. Proof -> completed.json
[ralph] git tag: phoenix-ralph-20260610-151719
driver exit=0   (driver writes the proof bundle + tag, not the agent)

Mechanism: a check is identified by a canonical digest recorded identically in the MCP path, CLI path, and ledger; accept proves that exact check went red→green on an intact hash-chained trace and is green now. The agent proposes state changes; the driver decides — and owns the loop/wall-clock/no-progress budgets, the sentinel, and the tag.
Honest limits: accept is only as strong as the check (hence command_exit-only top-level gates + failure-first); the driver bounds cost via budgets, not a hard sandbox; a live end-to-end Copilot build under the loop is the next evidence step. Grounded in Huntley's Ralph, Anthropic's Building Effective Agents + SWE-bench, BabyAGI, ReAct (research/autonomous-workflows-research.md).
